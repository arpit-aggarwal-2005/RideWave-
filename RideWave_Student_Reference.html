Why we built this pipeline
Every Feature Solved a Real Problem
RideWave had real business problems. Each Databricks feature you are learning was the solution. Read this before each lab — then the code will make sense.
PROBLEM 01 · DATA STORAGE
🗂️ All data in Excel and MySQL
RideWave's ops team could not answer "how many rides happened yesterday in Mumbai?" without exporting Excel sheets and counting manually. Data was scattered, stale, and unreliable.
Solution
We built a Data Lake on S3. All data lands here first. One place. Always available. Never deleted.
AWS S3 · Data Lake
PROBLEM 02 · RAW DATA QUERIES
📊 Business needed answers NOW
The full ETL pipeline takes weeks to build. But management needed ride counts and revenue figures immediately. They could not wait for the pipeline to be ready.
Solution
Athena queries raw CSV files in S3 directly using SQL. No database needed. No loading. Answers in minutes on raw data.
AWS Athena · SQL on S3
PROBLEM 03 · DATA QUALITY
🐛 Nulls and wrong types in raw data
fare_amount was stored as text ("800.50") not a number. 25 rides had null fare_amount. ride_status had mixed case: "Completed", "COMPLETED", "completed" — all treated as different values.
Solution
Glue ETL and Silver layer applies cleaning rules: cast types, drop nulls, standardise case. data_profiler.py finds the problems first so we know what to fix.
Glue ETL · Silver MERGE
PROBLEM 04 · AUDIT TRAIL
🔍 "Show me last week's data"
The finance team asked: "What did our revenue look like on March 15th?" But the pipeline was overwriting data every night. Yesterday's data was gone forever. No way to answer the question.
Solution
Bronze layer uses mode('append') NEVER overwrite. Every day's data is preserved. 4 metadata columns track WHEN and WHERE each record came from.
Bronze · Append-only · Metadata
PROBLEM 05 · DUPLICATES
💥 Pipeline ran twice — revenue doubled
The pipeline crashed at 2 AM and auto-restarted. It reprocessed all records. Every order now appeared twice. Finance reported double revenue to investors. Stock price moved on wrong data.
Solution
Silver uses MERGE not overwrite. MERGE checks: if ride_id exists → update. If new → insert. Run it 10 times — same result. Zero duplicates. This is called idempotency.
Silver · MERGE · Idempotency
PROBLEM 06 · REAL-TIME DATA
⚡ Ops needed live ride counts
The operations team needed to see rides happening RIGHT NOW — not yesterday's batch. They needed to know if Mumbai had enough drivers at 8 PM peak hour. Nightly batch was 12 hours too late.
Solution
Kinesis captures ride events in real-time. Firehose delivers to S3 every 60 seconds. Auto Loader picks up new files instantly. Data is live within 2 minutes of a ride completing.
Kinesis · Firehose · Auto Loader
PROBLEM 07 · FILE TRACKING
🔄 Auto Loader reprocessing old files
Every time the pipeline ran, Auto Loader read ALL files in S3 — including the ones already processed. Bronze grew 10x every day. Gold showed 10x revenue. All because old files were re-read.
Solution
Checkpoint is Auto Loader's memory. It saves which files were already processed to S3. Next run — skips old files, reads only new ones. Exactly-once guarantee.
Auto Loader · Checkpoint
PROBLEM 08 · SLOW QUERIES
🐌 Dashboard took 45 minutes to load
After 3 months of streaming, S3 had 50,000 small files. Spark opened each file separately — 50,000 open operations. The Gold dashboard query took 45 minutes. Users stopped using it.
Solution
OPTIMIZE compacts 50,000 small files into 5 large files. ZORDER BY (city) clusters city data together. Same query now runs in 2 minutes. Dashboard is usable again.
OPTIMIZE · ZORDER · Small file fix
PROBLEM 09 · BAD PIPELINE RUN
😱 Wrong data written to Gold table
A bug in the Gold aggregation wrote incorrect city revenue figures. The dashboard showed Mumbai revenue as ₹0. By the time it was caught, the original correct data was overwritten and gone.
Solution
Delta Time Travel. Every write creates a version. RESTORE TABLE gold_city_revenue VERSION AS OF 3 — rolls back to before the bug in seconds. Data is never truly lost.
Delta Time Travel · Version History
PROBLEM 10 · MANUAL PIPELINE
😴 Engineer runs pipeline at 6 AM daily
Someone had to wake up every morning at 6 AM to manually run Bronze → Silver → Gold notebooks. If they overslept or were sick — no data for the business that day. Not scalable.
Solution
Databricks Workflow. Bronze → Silver → Gold runs automatically every morning at 6 AM. If Bronze fails, Silver and Gold are skipped. Email alert sent. Engineer sleeps in.
Databricks Workflows · DAG · Schedule
PROBLEM 11 · DATA ACCESS
🔓 Everyone could see customer phone numbers
An intern analyst queried the customer table and saw full phone numbers and credit card numbers for all customers. GDPR violation. Legal team panicked. Company fined ₹2 crore.
Solution
Unity Catalog column masking. Analysts see ****-****-****-1001. Finance team sees full numbers. Same table, different views, controlled by role. One policy covers all tools.
Unity Catalog · Column Masking · GDPR
PROBLEM 12 · BUSINESS ANSWERS
📈 "Which city earns most? When are peak hours?"
The CEO asked: "Which city should we expand to next?" The ops head asked: "When should we send surge pricing alerts?" Nobody could answer from raw data — it was too complex to query directly.
Solution
Gold layer. Pre-aggregated tables: gold_city_revenue, gold_driver_performance, gold_peak_hours. CEO opens dashboard — answers are there. No SQL needed. Updated every morning automatically.
Gold Layer · CTEs · Business Answers

What Breaks Without This Feature
When an SME asks "why are you using this?" — this is your answer. Learn the break scenario, not just the feature name.
Feature What breaks without it Your answer to SME
mode('append') Bronze Layer
Yesterday's data is deleted every run
Finance asks "show last month's rides" → impossible. Regulatory audit needs historical data → company fined. No way to reprocess from source.
"Bronze is our audit trail. Overwrite destroys history. Append means every run adds a new partition. We can always go back to any day's data — even from 2 years ago."
4 Metadata Columns Bronze Layer
Cannot debug where bad data came from
Gold shows wrong revenue. You check Silver → Bronze → but which file caused it? Which pipeline run? Without _source, _ingest_ts, _file_name, _run_id — you are blind.
"These 4 columns are the audit trail inside each record. When data is wrong, I can trace it back to the exact source file, exact pipeline run, and exact time it was ingested."
MERGE (Silver) Silver Layer
Pipeline crash → duplicates on restart
Pipeline fails halfway, restarts. With overwrite → entire table rewritten. With INSERT → same records added twice. Silver has 2x the records. Gold shows 2x revenue. Wrong decisions made.
"MERGE is idempotent. Run it once — 157 rows. Run it 10 times — still 157 rows. If the pipeline crashes and restarts, it is safe. No duplicates ever. This is the production standard."
Checkpoint Auto Loader
Every run reprocesses all S3 files
Day 1: 100 files processed → Bronze has 300 rows. Day 2: pipeline re-reads all 100 + 10 new = 110 files → Bronze has 630 rows. After 1 month → Bronze has 10x data. Gold shows 10x revenue.
"Checkpoint is Auto Loader's memory. It writes processed file names to S3 after each run. Next run — skips already-processed files, reads only new ones. Exactly-once ingestion guaranteed."
Watermark Structured Streaming
Spark holds ALL window states in memory → cluster crash
Without watermark, Spark never cleans up old window state. After 24 hours of streaming with 5-minute windows → 288 windows × thousands of drivers → memory exhausted → cluster crashes → no data.
"Watermark tells Spark: wait max 10 minutes for late data, then close the window and free that memory. Without it, memory grows forever. With it, Spark can run for months without crashing."
OPTIMIZE + ZORDER Gold Layer
Thousands of small files → 45-minute dashboard load
3 months of streaming = 50,000 micro-batch files. Spark opens each file separately = 50,000 operations. Simple Gold query takes 45 minutes. Dashboard is unusable. Business stops trusting data.
"OPTIMIZE compacts 50,000 files into 5 files. ZORDER BY (city) clusters city data together so city-based queries scan less. Same query goes from 45 minutes to 2 minutes."
Time Travel Delta Lake
Bad pipeline run → correct data lost forever
A bug writes wrong revenue to Gold. Dashboard shows Mumbai revenue as ₹0. Without time travel — original correct data is overwritten. To fix, you must rerun the entire pipeline. Hours of downtime.
"Every Delta write creates a version. RESTORE TABLE gold_city_revenue VERSION AS OF 3 rolls back in seconds. No pipeline rerun needed. This is why Delta is production-safe — data is never truly lost."
Workflow DAG Orchestration
Pipeline runs on bad data → wrong Gold output
Without dependencies: Bronze fails but Silver still runs on empty table. Gold aggregates nothing. Dashboard shows ₹0 revenue. Looks like business collapsed. CEO panics. Without DAG — cascading failures.
"The DAG dependency means: Silver only runs if Bronze succeeded. Gold only runs if Silver succeeded. One failure stops the cascade. One alert email. No wrong data reaches the dashboard."
Unity Catalog Governance
Intern sees all customer credit card numbers → GDPR breach
Without Unity Catalog — every Databricks user can query every table. One SQL query exposes all customer PII. In India: DPDP Act violation. In Europe: GDPR fine. Company reputation destroyed.
"Unity Catalog controls who sees what at the column level. Analysts see masked credit card (****-1001). Finance sees full number. One policy covers notebooks, SQL, dashboards, and ML — all tools at once."
CTE Syntax (Gold) Gold Layer
New engineer cannot understand or maintain the query
A 50-line nested subquery with no names is impossible to debug at 3 AM when the Gold table is wrong. Without CTEs — the query is a black box. With CTEs — each step has a name and purpose.
"CTEs give each transformation step a meaningful name. cleaned_rides → city_summary → final result. Anyone can read it top to bottom and understand what it does. Maintenance-friendly code is production-ready code."

1
Bronze Ingestion — Append + Metadata
Block 7
Raw rides data → Bronze Delta table. Append-only preserves history. 4 metadata columns = audit trail. Partition by date = fast date queries.
# Add 4 mandatory Bronze metadata columns
df_bronze = df_raw
  .withColumn("_source",    F.lit("s3_raw_rides"))
  .withColumn("_ingest_ts",  F.current_timestamp())
  .withColumn("_file_name",  F.lit(S3_PATH))
  .withColumn("_run_id",     F.lit(RUN_ID))
  .withColumn("ingest_date", F.current_date())

df_bronze.write.format("delta")
  .mode("append")            # NEVER overwrite
  .partitionBy("ingest_date") # one folder per day
  .saveAsTable(f"{YOUR_DB}.rides_bronze")
⚡ Never use mode('overwrite') on Bronze. Ever. If your trainer sees it — red flag.
2
Silver Cleaning — 7 Rules
Block 8
Bronze has nulls, wrong types, mixed case. Silver applies business rules. Only completed rides. Proper types for calculations. Clean text for consistent joins.
df_silver = df_bronze
  .filter(F.col("ride_id").isNotNull())       # Rule 1
  .filter(F.col("fare_amount").isNotNull())   # Rule 2
  .filter(F.col("ride_status") == "completed") # Rule 3
  .withColumn("ride_date",
    F.to_date("ride_date", "yyyy-MM-dd"))    # Rule 4
  .withColumn("fare_amount",
    F.col("fare_amount").cast("double"))      # Rule 5
  .withColumn("ride_status",
    F.lower(F.col("ride_status")))           # Rule 6
  .withColumn("processing_date",
    F.current_date())                           # Rule 7
⚡ Bronze: 300 rows. Silver: ~157 rows. Dropped 143 = nulls + non-completed. Know this number.
3
MERGE — Idempotent Silver Write
Block 8
Run once = 157 rows. Run 10 times = still 157 rows. MERGE checks ride_id: exists → update, new → insert. Pipeline crash + restart = safe. No duplicates ever.
from delta.tables import DeltaTable

if not spark.catalog.tableExists(f"{YOUR_DB}.rides_silver"):
    df_silver.write.format("delta").mode("overwrite")
      .saveAsTable(f"{YOUR_DB}.rides_silver")
else:
    target = DeltaTable.forName(spark, f"{YOUR_DB}.rides_silver")
    (target.alias("tgt")
      .merge(df_silver.alias("src"),
             "tgt.ride_id = src.ride_id")  # match key
      .whenMatchedUpdateAll()                # SCD Type 1
      .whenNotMatchedInsertAll()             # insert new
      .execute()
    )
⚡ MERGE = SCD Type 1. whenMatchedUpdateAll replaces old with new. No history kept on matched rows.
4
Gold CTE — City Revenue
Block 9
Always read Gold from Silver — never Bronze. CTEs make the query readable. Each step has a name. New engineer can understand in 2 minutes. Production-ready code.
spark.sql(f"""
WITH cleaned_rides AS (
    SELECT * FROM {YOUR_DB}.rides_silver
    WHERE fare_amount IS NOT NULL
      AND ride_status = 'completed'
),
city_summary AS (
    SELECT city,
           COUNT(*)                   AS total_rides,
           ROUND(SUM(fare_amount), 2) AS total_revenue,
           ROUND(AVG(fare_amount), 2) AS avg_fare
    FROM cleaned_rides
    GROUP BY city
)
SELECT * FROM city_summary
ORDER BY total_revenue DESC
""").write.format("delta").mode("overwrite")
  .saveAsTable(f"{YOUR_GOLD_DB}.gold_city_revenue")
⚡ Gold uses overwrite (not MERGE) because it is fully recalculated every run from Silver. Aggregations are not incremental.
5
OPTIMIZE + ZORDER
Block 9
After many writes, Delta has thousands of small files. Spark opens each separately = slow. OPTIMIZE compacts them. ZORDER clusters by city so city queries skip irrelevant files.
# Run after every Gold table creation
spark.sql(f"""
    OPTIMIZE {YOUR_GOLD_DB}.gold_city_revenue
    ZORDER BY (city)
""")

# Measure the impact
before = spark.sql(
  f"DESCRIBE DETAIL {YOUR_GOLD_DB}.gold_city_revenue"
).select("numFiles", "sizeInBytes").first()
print(f"Files: {before['numFiles']}")
⚡ OPTIMIZE = fewer files = faster queries. ZORDER = data clustering = skip irrelevant files. Both together = maximum query speed.
6
Time Travel
Block 9
Every Delta write creates a version automatically. Query any past version. Restore if a bad pipeline run corrupts data. No data is ever truly lost in Delta Lake.
# See all versions
spark.sql(f"""
    DESCRIBE HISTORY {YOUR_GOLD_DB}.gold_city_revenue
""").display()

# Query version 0 (first write)
spark.sql(f"""
    SELECT * FROM {YOUR_GOLD_DB}.gold_city_revenue
    VERSION AS OF 0
""").display()

# Restore to a good version if pipeline went wrong
# RESTORE TABLE gold_city_revenue VERSION AS OF 3
⚡ Time Travel is Delta's superpower. Parquet has no versions. Delta remembers every write. This is why companies migrate from Parquet to Delta.
7
RANK() Window Function
Block 8 + 9
RANK() assigns position within a group. "Who is the #1 driver in Mumbai?" without RANK requires a self-join. With RANK = one clean query. PARTITION BY resets rank per group.
spark.sql(f"""
WITH driver_counts AS (
    SELECT driver_id, city, COUNT(*) AS ride_count
    FROM {YOUR_DB}.rides_silver
    GROUP BY driver_id, city
),
ranked AS (
    SELECT *,
        RANK() OVER (
            PARTITION BY city        -- reset rank per city
            ORDER BY ride_count DESC -- highest first
        ) AS city_rank
    FROM driver_counts
)
SELECT * FROM ranked
WHERE city_rank <= 3   -- top 3 per city
""").display()
⚡ RANK vs ROW_NUMBER: RANK gives same rank to ties (1,1,3). ROW_NUMBER always unique (1,2,3). DENSE_RANK no gaps (1,1,2).
8
Broadcast Join
Block 8
Joining rides (big) with drivers (small, 100 rows). Normal join = shuffle rides across network = slow. Broadcast = copy drivers to ALL executors = local join = fast. No network shuffle needed.
from pyspark.sql.functions import broadcast

df_rides   = spark.table(f"{YOUR_DB}.rides_silver")
df_drivers = spark.table(f"{YOUR_DB}.drivers_silver")

# Broadcast the SMALL table (100 rows)
result = df_rides.join(
    broadcast(df_drivers),  # sent to all executors
    "driver_id",
    "left"
)

result.explain(True)  # look for BroadcastHashJoin
⚡ Rule: broadcast tables under 10MB. Never broadcast large tables — OOM error. drivers (100 rows) = safe. rides (millions) = never.

Bronze Validator Function
Block 7
Before Silver runs, validate Bronze meets 4 quality standards. Catch issues early — better than discovering bad data in Gold when SME is watching your demo.
def bronze_validator(df, table_name, key_col, cols):
    row_count  = df.count()
    null_count = df.filter(F.col(key_col).isNull()).count()
    null_pct   = null_count / row_count * 100
    missing    = [c for c in cols if c not in df.columns]
    meta_cols  = ["_source","_ingest_ts","_file_name","_run_id"]
    meta_miss  = [c for c in meta_cols if c not in df.columns]

    print(f"  rows    : {row_count}")
    print(f"  nulls   : {null_count} ({null_pct:.1f}%)")
    print(f"  missing : {missing}")
    print(f"  metadata: {meta_miss}")

    return {
        "row_count": "pass" if row_count > 0 else "fail",
        "null_pct" : "pass" if null_pct < 5 else "fail",
        "schema"   : "pass" if not missing else "fail",
        "metadata" : "pass" if not meta_miss else "fail"
    }
⚡ In production — assert on validator results. If row_count == 'fail', raise Exception and stop the pipeline before bad data reaches Silver.

Recovery Cell — Every Notebook
All Blocks
Every notebook session is isolated. Cluster restart, compute switch, or opening a new notebook — all variables are gone. Recovery Cell makes each notebook self-contained and Workflow-ready.
# ── RECOVERY CELL — Run FIRST every session ──────────
YOUR_NAME   = "yourname"    # ← CHANGE THIS

if YOUR_NAME == "yourname":
    raise Exception("Change YOUR_NAME first!")

CATALOG     = "de_workspace26"
YOUR_DB     = f"{CATALOG}.ridewave_{YOUR_NAME}"
S3_PATH     = f"s3://se-de-q1-26/day8/{YOUR_NAME}/"
CKPT_BRONZE = f"{S3_PATH}checkpoints/bronze/"
CKPT_SILVER = f"{S3_PATH}checkpoints/silver/"
CKPT_GOLD   = f"{S3_PATH}checkpoints/gold/"

spark.sql(f"USE CATALOG {CATALOG}")
spark.sql(f"CREATE SCHEMA IF NOT EXISTS ridewave_{YOUR_NAME}")
print(f"✅ Ready — {YOUR_NAME} — {YOUR_DB}")
⚡ Variables do NOT travel between notebooks. Bronze notebook variables are gone when Silver opens. That is why EVERY notebook has its own Recovery Cell.
